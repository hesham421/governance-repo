**Major discrepancy vs. what CORE.md/the PLAN-SEC-002 memory assumed:** this repo's migration mechanism is no longer "Hibernate `ddl-auto` + hand-written `db/scripts/00N_*.sql`, DBA-run manually" — that was true as of PLAN-SEC-002's Phase CORE (2026-07-09) but has since been replaced. `erp-main` now owns a real, centralized Flyway pipeline (`spring.flyway.enabled=true`, `classpath:db/migration`, `V1`/`V2` already present — `V1` is literally a schema+data dump of the live DB captured via the Postgres MCP tool, `V2` is a hand-written incremental seed). `ddl-auto` is still `update` in dev / `none` elsewhere, unchanged. Followed the new convention (lowercase identifiers, `IF NOT EXISTS`, single versioned file per change) rather than db-script.md's literal uppercase `db/scripts` framing — Postgres folds unquoted identifiers to lowercase regardless, so the entity Java annotations (`FILE_CATEGORY`, `SEQ_FILE_CATEGORY`, etc., matching db-script.md's exact declared names) and the migration's lowercase DDL resolve to the same physical objects.

Deviations from db-script.md (all found by actually compiling/running against the live schema, not guessed in advance):

1. **`FILE_DOCUMENT.FILE_CONTENT` is nullable, not `NOT NULL` as db-script.md's own DDL literally states.** That DDL directly contradicts db-script.md's own column comment and RULE-FILE-006/OQ-001 (both require the application layer to null out `fileContent` on permanent delete while the row is retained) — a `NOT NULL` column cannot support that documented behavior. Made it nullable; the entity field/column carries a comment explaining the contradiction rather than silently picking a side.
2. **`IS_ACTIVE_FL` uses `INTEGER`, not db-script.md's stated `SMALLINT`** — matches this repo's actual `BooleanNumberConverter` mapping (`Boolean` → `Integer`), the same type every other boolean flag in the live schema already uses (`org_legal_entity.is_active_fl`, `roles.is_active`) — not a new deviation specific to this module, just following the platform's real convention over the plan's stated (and never-actually-used-anywhere) `SMALLINT`.
3. `FILE_DOCUMENT.FILE_CATEGORY_FK` keeps db-script.md's literal column name (no `_ID_FK` suffix) rather than the `create-entity` skill's generic `A.1.5` pattern — same precedent already live in this codebase (`OrgRegion.LEGAL_ENTITY_FK`), so not flagged as a new violation.
4. `application-dev.properties`'s `DB_URL` fallback default points at port `5433`, but the actual running Postgres container (`docker-compose.yml`, and this project's own `.mcp.json`) is on `5432` — discovered only because the first verification run failed with a connection refusal. Worked around via a `-DDB_URL` override for this verification run only; did **not** edit the shared properties file, since it's outside this module's ownership and out of DATAOM's scope — flagging for whoever owns `erp-main`'s shared config.
