Deviations from db-script.md (found by actually compiling/running against the live schema):

1. **`IS_ACTIVE_FL`/`IS_ENABLED_FL` use `INTEGER`, not db-script.md's stated `SMALLINT`** — matches this repo's actual `BooleanNumberConverter` mapping (`Boolean` → `Integer`), the same convention every other boolean flag in the live schema already uses (`file_category.is_active_fl`, `org_legal_entity.is_active_fl`) — same non-new deviation already flagged in V3 (FileSvc). `RETRY_COUNT` keeps `SMALLINT` unchanged — that one is a deliberate CORE.md/DRV-NOTIF-001 call (a count, not a flag), not subject to this convention.
2. **`CREATED_AT`/`UPDATED_AT`/`SENT_AT` use `Instant` in the entities, not `LocalDateTime`** as DATAOM.md's plan-language literally states — matches the platform's actual, consistently-used timestamp type everywhere else in this codebase (`AuditableEntity.createdAt`, `RefreshToken.expiresAt`, `PasswordResetToken.expiresAt` are all `Instant`; zero `LocalDateTime` usage found anywhere in any entity). Following the live convention over the plan's stated (and never-actually-used) type, same category of deviation as V3's `IS_ACTIVE_FL` finding.
3. `RECIPIENT_ID` is a plain `Long` scalar, not a JPA association — `erp-notification` has no Maven dependency on `erp-security` (same cross-module scalar-FK pattern as `SecUserProfile.branchIdFk`/`SecRoleBranch` → ORG). The FK (`FK_NOTIF_LOG_USERS` → `USERS.USERS_PK`) is still enforced live at the DB level, a documented Security EXCEPTION per DATAOM.md, not an XM-ID.
4. Same `DB_URL` port mismatch as FileSvc's own DATAOM handoff: `application-dev.properties` fallback default is port `5433`, live container is `5432`. Worked around via `-DDB_URL` override for this verification run only; did not edit the shared properties file (outside this module's ownership).
