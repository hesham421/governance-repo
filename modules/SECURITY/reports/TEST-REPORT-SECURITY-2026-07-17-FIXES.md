## Environment drift since 2026-07-11 (found before any code fix)

The environment had moved on in the 6 days since the baseline report, independent of anything in
this session's diff:

1. **`users`/`permissions`/`roles` identity-sequence desync** (real, high-impact bug, fixed via SQL):
   Postgres `GENERATED BY DEFAULT AS IDENTITY` columns backing these 3 tables had their sequences
   lagging far behind `MAX(pk)` — `permissions_permissions_pk_seq` was at 72 vs. max 229,
   `roles_roles_pk_seq` at 5 vs. max 17 (both almost certainly from an initial seed script that used
   explicit PKs without ever calling `setval()`). This silently caused sporadic `DataIntegrityViolationException`
   (`duplicate key value violates unique constraint`) once organic test traffic's sequence-driven
   inserts caught up to already-occupied PK values — this is what broke `self-service-auth.spec.ts`'s
   signup baseline calls (`TC-SSA-002/003/005/016`, expected 200 got 409) before any other fix was
   applied. Fixed via `SELECT setval('<seq>', (SELECT MAX(<pk>) FROM <table>))` for all 4
   identity-backed tables (`users`, `permissions`, `roles`, `refresh_tokens` — the last was already
   in sync but re-synced for safety).
2. **New Group 11 (e2e-flows) failures**: baseline reported 2 fails; a fresh run (before any fix)
   showed 6 (`TC-E2E-002/003/005a/007/008/009`), all downstream of the same bugs below or the sequence
   issue — none were a new, independent regression once the underlying bugs were fixed.
3. **`user-profiles.spec.ts`**: baseline's 9 SKIPPED were `DB_PRECONDITION` (missing
   `PERM_USER_PROFILE_*` grant, tracked in `PLAN-SEC-002 Phase 3` memory). That grant was already
   applied 2026-07-15 (before this session) — confirmed live via `mcp__postgres__query`. Re-running
   these tests now exercises real functionality instead of a permission wall; found & fixed 3 further
   real bugs surfaced by the previously-untested UI paths (see below).

## Bugs fixed (backend/frontend source, not test-only)

| # | Component | Fix |
|---|---|---|
| 1 | `PagesFormComponent` (frontend) | `updatePage()` sent `active` in the PUT body; `UpdatePageRequest` (backend) has no such field and `fail-on-unknown-properties=true` global rejects it as malformed JSON. Removed `active` from the update payload (and the TS interface) — activate/deactivate already has dedicated endpoints. |
| 2 | `CreateUserRequest` (backend) | Real Add User drawer always sends `enabled`; the record only had `username`+`password`. Added an optional `enabled` field (defaults to `true` when absent, preserving prior behavior for any other caller), and `UserService.createUser` now respects it. |
| 3a | `RoleBranchApiService`/`UserProfileApiService` (frontend) | Both sent `field: 'isActive'` to the org-branches search endpoint; only `isActiveFl` is a valid search field there. Fixed both call sites — this alone also fixed 3 previously-unrelated UI test hangs (`TC-RB-001/008/010`, `TC-UP-001/002/007/008`) whose branch dropdowns had been silently empty. |
| 3b | `UserFacade` (frontend) | `convertSpecFiltersToSearchFilters`/`convertAgGridFiltersToSearchFilters`/`convertCustomFiltersToSearchFilters` all sent `{field, op, value}`; backend's `BaseSearchContractRequest.ContractFilter` expects `operator`. Renamed the field everywhere it's produced (facade + `SearchFilter` interface + one template binding). |
| 4 | `NavCollapseComponent` (frontend) | **Not fixed the way the baseline report recommended.** The report suggested populating `MenuItemDto.permCode` on the backend — doing so actually made things *worse*: it fed a permission-code string (`PERM_X_VIEW`) into a client-side check (`item.role.includes(currentUserRole)`) that compares against a coarse ROLE NAME, which can never match a permission code — this broke leaf items too, not just parent/collapse ones. Root cause is actually `NavCollapseComponent.isEnabled` defaulting to `false` (only enabled on an explicit role match) while the sibling `NavItemComponent` defaults to `true` ("disabled only if role check fails") — and since the backend's menu is already server-side filtered to exactly what the user is permitted to see, no client-side re-check is meaningful here at all. Fixed by aligning `NavCollapseComponent`'s default to `true`, matching `NavItemComponent`; left `permCode` unset as before. |
| 5 | `PermissionService.searchPermissions` (backend) | `module` was in the search whitelist but `Permission` has no such column (it's on the related `Page`); filtering by it threw `500`. `pageId`/`pageCode` were rejected outright. Now remapped internally to `page.module`/`page.id`/`page.pageCode` before validation. |
| 6 | `SpecBuilder.getExpression` (backend, shared `erp-common-utils`) | Deeper bug behind #5: any dot-notation join through a `@ManyToOne`/`@OneToOne` singular association threw `ClassCastException` on this Hibernate version (a singular-association `Path` doesn't implement `From`), and the existing fallback only caught `IllegalArgumentException`/`IllegalStateException`. Added `ClassCastException` to the catch so it falls through to the join-based path like collection associations already do — a shared-utility fix, not permission-specific. |
| — | `NotificationBellComponent` (frontend) — **found this session, not in baseline report** | The always-loaded notification bell calls `/api/v1/notifications/history/search` on every page for every user, unconditionally. A user without `PERM_NOTIFICATION_INBOX_VIEW` (only `SUPER_ADMIN` has it) got a `403`, and the global `ErrorInterceptor` treats **any** `403` from **any** call as cause to redirect the whole app to `/access-denied` — silently locking such a user out of every page, not just the bell. This is what broke `TC-MENU-001/002` once a narrow-permission throwaway test role was used. Fixed by guarding the bell's API call behind `permissionService.hasPermission('PERM_NOTIFICATION_INBOX_VIEW')` — user declined (see below) to also grant this permission to the default `ROLE_USER` role, so the bell now silently stays empty for non-admins rather than crashing the app; this is an explicit, confirmed decision, not an oversight. |
