CLIENT-SPECIFIC POLICIES
──────────────────────────────────────────────────────────────────
No policies were stated in vision.md specifically for File Service.
Entries below originate from the ARCH-REF-1.10 session input — marked
"Source: ARCH-REF" so P1 can distinguish them from vision-text-derived
policies.

POLICY-CLI-01: File size limits (two-tier)
  Rule   : The system MUST reject any HTTP request exceeding 10MB
           (transport limit) and any file content exceeding 5MB
           (business limit), independently enforced. Per-category
           overrides are allowed via FileCategory.max_size_bytes.
  Trigger: On upload
  Source : ARCH-REF AD-FILE-05 / RULE-FILE-UPLOAD

POLICY-CLI-02: Encrypted Token requirement for all file operations
  Rule   : Every upload/download/delete operation MUST be authorized via
           a single-use, time-bound Encrypted Token (AES/GCM, 12-byte IV,
           128-bit GCM tag) embedded in the URL path — not via
           Authorization header. TTL = 100 minutes (configurable). A
           request with a missing, tampered, expired, or action-mismatched
           token MUST be rejected (401/403) before reaching business logic.
  Trigger: On every request to /upload/{token}, /download/{token},
           /{token} (delete)
  Source : ARCH-REF AD-FILE-02 / RULE-FILE-DOWNLOAD

POLICY-CLI-03: Token generation is File Service's responsibility
  Rule   : The Encrypted Token is generated by File Service itself (via
           a dedicated token-issuing endpoint) and returned to the
           caller (Frontend/Mobile) — the caller does not construct its
           own token. This is an explicit addition over the reference
           architecture, which assumed the caller creates the token.
  Trigger: On upload/download intent, before the actual file operation
  Source : ARCH-REF P1 guidance note (API-FILE-001 addition)

POLICY-CLI-04: Deletion is final; ownership-restricted
  Rule   : Deletion MUST be permanent — no Recycle Bin in Phase 1. Only
           the file's owning entity's authorized actor or an Admin role
           may delete a file; a delete attempt from an unrelated
           entity/module MUST be rejected with 403.
  Trigger: On delete
  Source : ARCH-REF RULE-FILE-DELETE

POLICY-CLI-05: MIME type detected server-side, not trusted from client
  Rule   : The system MUST detect MIME type from file content itself,
           never trust a client-supplied Content-Type header for
           validation decisions.
  Trigger: On upload
  Source : ARCH-REF AD-FILE-06

POLICY-CLI-06: No JWT validation inside the module
  Rule   : File Service MUST NOT independently validate JWT tokens for
           its own endpoints; the Encrypted Token is a separate,
           purpose-built security mechanism. JWT trust (for the
           token-issuing endpoint itself) is delegated to Security's
           filter chain.
  Trigger: On any incoming request
  Source : ARCH-REF ADAPT-03

CUSTOM LOV VALUES
──────────────────────────────────────────────────────────────────
None — File Service defines no MD_LOOKUP_DETAIL-governed values.
FileCategory rows (module_code-scoped) are Reference Table data, not
Lookup values — each consuming module (Procurement, Notification, etc.)
seeds its own category rows (e.g. "ATTACHMENT", "REPORT",
"NOTIFICATION_TEMPLATE") independently, without touching this module's
code.

SCOPE EXCEPTIONS
──────────────────────────────────────────────────────────────────
Excluded (per ARCH-REF Section 5 — "WHAT DOES NOT APPLY"):
  Eureka service discovery   : N/A — platform is a Modular Monolith
  heac-util-client (Feign)   : replaced by direct @Service injection
  heac-genericlibrary        : replaced by platform's own AuditableEntity
  Multi-schema (heac/heac_util): N/A — single PostgreSQL schema
  JWT validation inside module : owned by Security (1.2)

Excluded permanently (this session — not deferred):
  PDFBox / PDF processing    : no defined use case even in the source
    reference; excluded entirely rather than carried as an open item.
  RabbitMQ / async file ops  : core operations are synchronous by
    nature; direct method calls used instead across the monolith.

Deferred (technical, activation trigger noted):
  Migration BYTEA → Large Objects : only relevant if a future file-size
    ceiling increase (e.g., multi-GB attachments) makes BYTEA
    impractical. No current trigger — file size cap is 5MB.

Confirmed out of scope for this module (ownership boundary):
  Deciding which categories a module needs — owned by each consuming
    module (Procurement/Sales/.../Notification), which seeds its own
    FileCategory rows; File Service does not hardcode categories.
  JWT/session authentication — owned by Security (1.2)
  Deciding whether a document requires OCR/preview/validation beyond
    MIME-type detection — out of scope until a concrete need is raised

PLATFORM INTEGRATION NOTES
──────────────────────────────────────────────────────────────────
Database target:
  DB_TARGET = POSTGRESQL_16. File content stored as BYTEA (not Oracle
  BLOB, not PostgreSQL Large Objects). P2 applies CORE-8 PostgreSQL
  type mapping to FILE_DOCUMENT / FILE_CATEGORY (BIGINT PKs, VARCHAR,
  SMALLINT for flags).

Connection pooling:
  Standard Spring Boot HikariCP — Oracle UCP guidance from the source
  reference does not apply.

Integration pattern:
  Direct @Service injection from every consuming module — no RabbitMQ
  in the core path (see module-registry-file.md AUTO-DECISIONS for the
  reasoning). The Encrypted Token mechanism is exclusively for
  browser/mobile-facing URLs (Angular FileUploadComponent, Flutter),
  not for inter-module calls.

Notification Service integration (forward-looking):
  This module is the target of Notification's DEFERRED XM dependency
  (XM-NOTIF-[TBD], module-registry-notif.md). When this module's DBS-ID
  gates at MODE 1.5, the Registry Maintainer fires RXE-NOTIF-[SEQ] per
  CONTRACT-8, and Notification's own P3 session migrates template
  bodies from inline storage to FILE_DOCUMENT. No action required from
  File Service's own pipeline beyond existing normally — this is
  Notification's migration to perform, not File Service's.

Secrets management:
  DOCUMENT_ENCRYPTION_SECRET (AES key) MUST be injected from an
  external environment file outside the build (ARCH-REF AD-FILE-09) —
  infrastructure/secrets concern, out of P0 architectural scope, noted
  here so P1/P3 do not treat it as a missing requirement.
