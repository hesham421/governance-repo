## Deviations / judgment calls (not silent)

1. **STEP 0 CONTRACT ALIGNMENT GATE was force-overridden by explicit user instruction.** At the time this phase started, `execution-state.json` had `current_phase: DOC` (PENDING) and `INT-C` also PENDING — the gate (plan Section 12) blocks F1 until both are COMPLETE, and the phase file's own SESSION PROMPT additionally requires `HANDOFF-PHASE-6-INT-R.md` to exist (it doesn't). I stopped and asked before writing any code; the user explicitly chose "force-start F1 anyway." Proceeding on that basis — this is a known, deliberate governance-order violation, not an oversight. DOC/INT-C/INT-R remain PENDING in `execution-state.json` and should be executed and reconciled before F1's work is trusted as being built on a stable, aligned contract.
2. **SCR-SEC-006 has no facade/service/model files** — deliberately. F2 owns `UserProfileFacade` (plan Section 7.2); building it now would duplicate/conflict with that phase's ownership. The search/entry components are structural shells only (`ErpEmptyStateComponent` placeholder, no AG Grid, no Reactive Form wired to a backend) per the phase file's own "no business logic" instruction. Confirms with a `create-components`/`create-routing` skill note: those skills assume facade-first ordering ("When NOT to Use: before facade is defined") which doesn't hold for this flat-phase plan — routes were still built following the skills' structural pattern (3 routes, `authGuard+permissionGuard`, `AdminLayout`), just without the facade-dependent parts (confirm-actions helper, AG Grid wiring).
3. **SCR-SEC-002 Branch Scope tab is a placeholder** (`ROLE_ACCESS.BRANCH_SCOPE_COMING_SOON` message) — `RoleBranchFacade` is also F2-owned (Section 7.2). The Save/Copy-From/Add-Page action-bar buttons are now scoped to only show on the "pages" tab (they don't apply to Branch Scope) — a necessary follow-on to the tab addition, not separately requested but required for the UI to not misrepresent what Save does. Fixed an Angular NG8011 content-projection warning this introduced by wrapping the two `actionBarEnd`-slotted buttons in an `<ng-container actionBarEnd>` rather than leaving them as direct siblings under a conditional `@if`.
4. **SCR-SEC-008/009 did NOT reuse the pre-existing `register`/`forgot-password` routes and components.** Those are unmodified leftover "CodedThemes" starter-theme scaffolding (hardcoded English text, no i18n, fake Google/Twitter/Facebook buttons, a legacy `authenticationService.register()` call with no activation step) that happened to share route paths the login screen already linked to — they don't implement this plan's actual contract (`SignUpFacade.submit()→API-SEC-040`/`activate(token)→API-SEC-041`; `ForgotPasswordFacade.requestReset()→API-SEC-042`/`reset(token)→API-SEC-043`). Per the DoD's literal "3 **new** routes" and "wired to their **new** destinations" wording, I built new dedicated routes/components instead and repointed the login links to them. The old `register`/`forgot-password` routes/components/redirects were left untouched (not deleted) — they're now unreferenced from the UI but still directly reachable by URL. Flagging as legacy/orphaned, candidate for cleanup once F2/F3 land, not removed here since deletion wasn't requested and out of scope for a routing/shell phase.
5. **Root `app-routing.module.ts` was touched** (`create-routing` skill explicitly flags root-routing changes as "a structural change requiring manual review"). Change is minimal and follows an exact existing precedent (`register`/`forgot-password` redirect lines) — two new redirect lines, nothing else.
